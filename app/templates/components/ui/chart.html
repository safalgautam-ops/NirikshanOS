{# Dependency-free charts for dashboard widgets - no vendored JS, just CSS height/width percentages and conic-gradient (same technique as progress.html/meter.html), so they work under this app's strict CSP without adding a charting library (e.g. recharts, which needs React - this app is server-rendered Jinja with plain JavaScript). Callers precompute `pct`/`start`/`end` (0-100) and `display` (pre-formatted label) per item in the service layer - these macros never do math on raw values, matching how every other macro here stays presentation-only. bar_chart/hbar_chart item shape: {"label": str, "display": str, "pct": number}. pie_chart slice shape: {"label": str, "display": str, "color": str (a CSS color/var()), "start": number, "end": number} (0-100 conic-gradient stops). line_chart series shape: {"points": [{"x": number, "y": number}, ...] (0-100, y already inverted/padded), "labels": [{"x": number, "text": str}, ...] (a sparser subset of points, so labels don't collide), "value_labels": [{"x": number, "y": number, "text": str}, ...] (just the turning points - local peaks/troughs plus both endpoints - each rendered as a dot + its real value floating above the line). bar_chart_labeled item shape: {"label": str, "display": str, "bar_pct": number} - like bar_chart/hbar_chart's `pct`, but floored to a legible minimum (see dashboard/service.py._bars_labeled) since the label renders inside the bar itself, not beside it - a true zero/near-zero value still needs a non-empty bar for its label to sit on. `display` (the real, unfloored value) always renders outside the bar, so nothing is misrepresented. radar_chart series shape: {"cx": number, "cy": number, "grid_rings": [str, ...] (each a ready-made SVG value), "spokes": [{"x2": number, "y2": number}, ...] (center-to-edge axis lines), "polygon_points": str (the data shape's own value), "dots": [{"x": number, "y": number, "display": str}, ...], "labels": [{"x": number, "y": number, "text": str}, ...] (outer axis category names) - all in the same uniform-scale 0-100 square, since the macro renders it inside an aspect-square container (unlike line_chart, real SVG / shapes here don't need the non-scaling-stroke workaround - x and y always scale by the same factor). radial_chart item shape: {"label": str, "display": str, "radius": number, "thickness": number, "hit_thickness": number, "dash": number, "gap": number, "color": str} (see dashboard/service.py._radial_bars - dash/gap are real SVG stroke-dasharray lengths for that ring's own circumference, not percentages, so each ring's arc is drawn with a plain + stroke-dasharray instead of a hand-built arc ). Each ring's label only appears on hover, via a plain CSS group/group-hover pair scoped to that one ring's own wrapper div - intentionally not a Tailwind *named* group (group/foo), since a per-index dynamic class name wouldn't survive a real (non-dev-JIT) Tailwind build; every ring reuses the same static "group"/"group-hover:" pair in its own independent wrapper instead.
