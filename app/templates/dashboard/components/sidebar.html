{# Sidebar - primary navigation. Usage: {% from "dashboard/components/sidebar.html" import sidebar %} {{ sidebar(active="dashboard", visible_keys=visible_keys) }} Built on top of components/ui/sidebar.html's macros (a Jinja port of shadcn/ui's Sidebar primitives - see that file's doc comment for how the desktop/mobile split works). This file owns the actual nav structure/gating rules; components/ui/sidebar.html only owns the reusable shell markup. Params: active - key of the current nav item, used to highlight it. visible_keys - which Administration keys this user's *system* role allows, or None for "no restriction" (see app/core/security/permissions.py NAV_KEYS - both lists must stay in sync). One Platform group (Dashboard, Organization, and - mixed directly in, no separate "Your Organization" heading - the org's own Staff/Roles), an Application group (org-scoped feature pages like Case, gated the same way as Staff/Roles), plus a separately-gated Administration group. Three independent rules, never mix these up: 1. Dashboard/Organization - always rendered for every logged-in user, never permission-gated and never locked. Baseline navigation plus the only path to unlock everything else. 2. The org's Staff/Roles items, listed right after Organization in the same group - filtered by g.org_nav_keys (set in load_session(), see app/core/security/sessions.py / org_permissions.py). Two states: - g.org_locked (no active org yet) - shown LOCKED. This is the one place locking is honest: joining/creating an org is exactly what unlocks it. - otherwise - hidden if the org role doesn't grant it, shown if it does. No lock icon past this point - access depends on what the org admin assigned, not on an action the user themself can take. 3. Administration (system pages) - filtered by visible_keys. Hidden (not locked) when the key isn't granted - most users hold zero system roles, so this group renders nothing for them. There's no "do X to unlock this" action available, so a lock icon would be misleading; it's just absent. Platform staff exception: anyone holding a system permission (g.is_platform_staff, also set in load_session()) never sees "Organization" or the org's Staff/Roles items at all, locked or not - they manage every org from /admin/organizations instead of being a tenant member through their own platform identity (see onboarding/routes.py's blueprint guard, which enforces this server-side too). #} {%
